# Аудит сайта VIPAUTO161 (структура, контент, дизайн, SEO)

Дата аудита: 03.03.2026  
Источник: анализ исходного кода проекта `vipauto_161` (Next.js App Router).

## 1) Краткое резюме

Сайт представляет собой одностраничный сервисный лендинг автоэлектрики в г. Шахты с двумя юридическими страницами (`/privacy`, `/terms`), техническими маршрутами SEO (`/sitemap.xml`, `/robots.txt`) и API-эндпоинтом заявок (`/api/lead`).

Главная цель сайта: конверсия в запись через:
- WhatsApp / Telegram (основной путь),
- телефонный звонок,
- форму заявки.

Позиционирование: премиальный центр автоэлектрики и автоэлектроники, официальный партнер StarLine.

## 2) Техническая платформа

- Framework: Next.js `16.1.6` (App Router).
- Frontend: React `19.2.3`, TypeScript strict mode.
- Styling: Tailwind CSS v4 + кастомные CSS-токены.
- Motion/animation: GSAP `3.14.2` + ScrollTrigger.
- Icons: `lucide-react`.
- Fonts: Google Fonts (Manrope + JetBrains Mono).
- Аналитика: Яндекс.Метрика (подключается при `NEXT_PUBLIC_YANDEX_METRIKA_ID`).
- Обработка лидов: API route, отправка в Telegram Bot API (если есть токен/chat id), fallback в WhatsApp.

## 3) Карта маршрутов (фактическая)

### Публичные страницы

- `/` — главная лендинг-страница.
- `/privacy` — политика конфиденциальности.
- `/terms` — публичная оферта.

### API

- `/api/lead` (POST) — прием заявок с формы.

## 6) UX и конверсионные сценарии

Основной путь:
1. Пользователь видит оффер в Hero.
2. Выбирает быстрый канал (WhatsApp/Telegram/звонок) или скроллит к форме.
3. Отправляет заявку через форму либо обращается в мессенджер.

UX-усилители:
- фиксированное мобильное меню,
- фиксированная мобильная панель быстрых действий (звонок/запись),
- контрастные CTA-кнопки,
- прозрачная структура «услуги -> преимущества -> процесс -> отзывы -> контакт».

Потенциальные UX-ограничения:
- много визуальных эффектов одновременно (видео + parallax + glow + GSAP), риск сенсорной перегрузки и падения FPS на слабых устройствах.
- nav-фокус в меню кастомный, но нет полноценного focus-trap внутри overlay-диалога.

## 8) SEO-аудит

## 8.1 Что реализовано хорошо

- Корректный `metadataBase`.
- Title/description на главной.
- canonical на `/`, `/privacy`, `/terms`.
- Open Graph + Twitter metadata.
- `robots` + `sitemap`.
- `lang="ru"` на `<html>`.
- JSON-LD LocalBusiness (`AutoRepair`) с адресом, телефоном, рейтингом, sameAs.
- Локальная семантика по контенту (город, услуги, StarLine, рейтинг).

## 8.2 Ограничения текущей SEO-модели

- Сайт почти полностью одноуровневый (1 коммерческая страница + 2 юридические).  
  Ограничивает охват long-tail запросов по отдельным услугам.
- Нет отдельных SEO-посадочных по услугам/маркам/проблемам.
- OG-изображение задано как SVG (`/images/plate-logo.svg`), что не всегда оптимально для сниппетов соцсетей.
- В sitemap `lastModified` = `new Date()` при генерации; значение «текущая дата» может меняться без фактических изменений контента.
- Нет отдельного блока FAQ + FAQPage schema.
- Нет breadcrumb schema.
- Нет hreflang (если нужна мультиязычность в будущем).

## 8.3 SEO-риски контента/техники

- На странице `app/not-found.tsx` текст отображается как «кракозябры» в исходнике (битая кодировка строк).  
  Для SEO это не критично по трафику, но влияет на качество бренда и индексацию страницы 404.
- В шапке/меню якорная навигация (`#...`) без полноценных внутренних URL-страниц не расширяет индексацию.

## 8.4 Приоритетные SEO-улучшения

P0:
- исправить кодировку текста в `not-found.tsx`;
- подготовить OG-изображение в PNG/JPG (`1200x630`) и подключить в metadata.

P1:
- добавить отдельные страницы услуг (`/services/starline`, `/services/diagnostics`, ...);
- сделать уникальные title/description/h1 под каждую услугу;
- добавить FAQ-блок на ключевых страницах + schema FAQPage.

P2:
- добавить статьи/кейсы по симптомам автоэлектрики (контент-маркетинг);
- расширить internal linking между услугами, кейсами и контактом.

## 9) Технический SEO и индексирование

- `robots.ts`: index/follow, sitemap + host выставлены.
- `sitemap.ts`: список URL корректный, но минимальный.
- Canonical задан для всех текущих страниц.
- Семантическая структура:
  - главная: 1 H1, далее H2/H3 по секциям.
  - legal-страницы: H1 + повторяемые H2.
- Нет явных признаков блокировки индексации.

## 10) Форма заявки и backend-поток

### Frontend (`LeadForm`)

- Валидация:
  - `name >= 2`,
  - `phone` по regex `^\+?\d{11,15}$`,
  - service обязателен через select.
- Состояния: idle/loading/success/error.

### API (`/api/lead`)

- In-memory rate limit: 3 запроса за 10 минут на IP.
- При отсутствии Telegram env — возвращает `ok: true` + fallback WhatsApp.
- При ошибке Telegram API — также мягкий fallback.

### Технические ограничения

- Rate limit в памяти процесса подходит для простого сценария, но слаб в serverless/горизонтальном масштабировании.
- Нет anti-bot (captcha/honeypot), теоретически возможно спам-нагрузка.

## 11) Юридический контент

Страница `/privacy`:
- 10 разделов: сбор/цели/основания/хранение/cookies/права субъекта/контакты.

Страница `/terms`:
- 12 разделов: предмет, акцепт, стоимость, гарантия, ответственность, споры, форс-мажор, реквизиты.

Плюсы:
- есть реквизиты ИП, ОГРНИП/ИНН/ОКПО/ОКВЭД,
- есть canonical и metadata для legal-страниц.

## 12) Неиспользуемые или легаси-компоненты (по текущему рендеру)

Найден отдельный старый контур навигации:
- `components/layout/Header.tsx` не подключен в `app/layout.tsx`.
- `components/navigation/*` используются только внутри этого Header, фактически не участвуют в текущем UI.

Дополнительно присутствуют компоненты, не участвующие в текущем рендере главной:
- `components/effects/ParticleField.tsx`
- `components/effects/ReverseParallaxBackground.tsx`
- `components/ui/AnimatedCounter.tsx`
- `components/parallax/ParallaxImage.tsx`
- `components/parallax/ParallaxParticles.tsx`

Это не ошибка, но увеличивает сложность поддержки и может запутывать при дальнейшем развитии.

## 13) Производительность и стабильность

Потенциальные узкие места:
- фоновое видео (`/uploads/videos/hader.mp4`) + множественные blur/particles/GSAP;
- большое количество client-компонентов;
- интенсивный scroll-driven motion.

Компенсации, которые уже есть:
- `prefers-reduced-motion` поддержан во многих местах;
- часть анимаций отключается/смягчается на мобильных.

Рекомендации:
- проверить Core Web Vitals (LCP/CLS/INP) в реальном проде;
- сжать/оптимизировать видеофайл и оценить poster/lazy-подход;
- ограничить число одновременных анимаций above-the-fold.

## 14) Сильные стороны проекта

- Четкая локальная коммерческая направленность.
- Сильная визуальная айдентика.
- Хорошо читаемая структура лендинга для сервисного бизнеса.
- Достаточно развитая UX-анимация и мобильный сценарий.
- Базовый технический SEO уже внедрен.
- Есть legal-база и структурированные данные LocalBusiness.

## 15) Ключевые зоны роста

1. Расширить SEO-структуру с одного лендинга до кластера посадочных страниц по услугам.
2. Усилить trust-контент (кейсы, до/после, FAQ, сертификаты, фото работ).
3. Упростить/оптимизировать визуальные эффекты для слабых устройств.
4. Убрать/архивировать легаси-компоненты, чтобы снизить техдолг.
5. Доработать обработку лидов (anti-spam, устойчивый rate-limit backend-уровня).
